# Decision rules & governance
**The single most important design choice: the AI assists, but a deterministic rule decides.**

## 1. What runs the decision — and what does not
- **The LLM (Ava) does NOT decide the outcome.** It reads the claim and returns *signals* only: claim type, severity, fraud-risk score (0–100), an FCA vulnerability flag, and a summary — as schema-constrained JSON.
- **A deterministic rule decides accept / refer / decline.** It is a small, explicit **JavaScript function inside the n8n workflow** (the "Build Claim Record" Code node). It is **not Python**, and it is **not the model**. It is plain code: a lookup table + `if / else` conditions.
- Because the decision is a pure function of its inputs — no model call, no randomness, no time dependence — **the same claim always produces the same outcome**, and the logic can be read and audited line by line.

## 2. The rule (exactly as it runs)
```
excess = { Motor: 250, Home: 150, Travel: 100, Liability: 250 }[incident_type]  (default 100)

if   value < excess                                   → DECLINE  → Legal     (status: declined)
else if fraud_risk >= 60                              → REFER    → Legal     (status: referred)
else if vulnerable_flag                               → REFER    → Adjuster  (status: referred)
else if value >= 5000  OR  severity == "High"         → REFER    → Adjuster  (status: referred)
else                                                  → ACCEPT   → Finance   (status: approved, payout = value)
```
- **Order matters and is intentional:** a below-excess claim is declined first (no payment is due); fraud is checked before everything else; vulnerability is protected before value/severity; only clean, in-scope, lower-risk claims are accepted straight through to Finance.
- **Standardised references** are generated by PostgreSQL triggers + sequences: claims `ALB-<TYPE>-<YYYY>-<seq>` (e.g. `ALB-MOT-2026-001049`); payments `PAY-<YYYY>-<seq>` (e.g. `PAY-2026-005002`).
- **Team mailboxes** are mapped in the same function: Claims Adjuster → send2chopstix@gmail.com · Legal → Nishachopra.uk@gmail.com · Finance → leeakpareva@gmail.com.

## 3. How a new rule is added or changed
1. **Edit the function.** Add a new excess, a new threshold, or a new routing condition in the decision Code node (e.g. raise the adjuster threshold from £5,000 to £7,500, or add a "Travel claims over £2,000 → Adjuster" branch).
2. **Review it in Git.** The workflow is exported to `workflow/*.json` and committed, so every rule change is a reviewable diff — attributed, dated and reversible. Nothing is hidden inside a prompt; no model retraining is involved.
3. **Deploy it.** Re-import the workflow into n8n; the new rule is live for the next claim. The previous version is one `git revert` away.

**Production next step (stated honestly):** lift the thresholds and excess table out of code into a `jeen.rules` config table (or a rules engine), so risk & compliance can propose changes through a formal review/sign-off flow without an engineer — while keeping the same deterministic, auditable evaluation.

## 4. Why deterministic matters here (and how it's guaranteed)
- **Reproducibility:** re-running the same claim yields the same decision — essential for audit, appeals and regulator scrutiny.
- **Explainability:** we can state precisely *why* a claim was accepted, referred or declined ("below the £250 motor excess"; "fraud score 72 ≥ 60"; "vulnerable customer").
- **No model drift in outcomes:** changing the model or its temperature cannot change a decision — the rule sits downstream of the model and consumes only structured fields.
- **Guarantees in code:** the decision function makes no network/model call and uses no `Math.random()` or time value; identical inputs → identical output.

## 5. Human-in-the-loop (who decides what)
| Outcome | Decided by | Then |
|---|---|---|
| Accept | the rule, **payment released by a human** (Finance) | Finance "pay" → status `paid` + payment reference |
| Refer | a **named team** (Claims Adjuster or Legal) | approve / decline / request-info / escalate / clear-fraud |
| Decline | the rule, **appeal owned by a human** (Legal) | claimant gets 14-day appeal rights; Legal can uphold the appeal |

No high-value (≥ £5,000), high-fraud-risk (≥ 60), or vulnerable-customer claim is ever actioned without a person.

## 6. Ethical AI on a sensitive process
Insurance decisions affect people at vulnerable moments. The build is governed by six principles (also shown on the app's Data & Security page):
1. **Human accountability** — every sensitive outcome is owned and actioned by a named person; the AI surfaces signals, people decide.
2. **Deterministic & explainable** — outcomes come from an explicit rule, not a black box; we can explain and reproduce any decision.
3. **Fairness & vulnerability** — an FCA Consumer-Duty vulnerability check runs on every claim; the same rule is applied to everyone, consistently — not at discretion.
4. **Privacy & honesty** — data minimised, secrets server-side; Ava never invents facts, gives advice, or promises a payout, is clearly identified as an AI, and always has a human to escalate to.
5. **Auditable & contestable** — every claim and action is timestamped in PostgreSQL; decisions can be reviewed, appealed and corrected.
6. **Bounded automation** — the model's tools are assistive and read-only; it cannot settle, pay or reject. The line between AI and human is explicit and enforced in the workflow.
